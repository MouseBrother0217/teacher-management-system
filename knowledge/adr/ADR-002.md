# ADR-002: 数据库持久化决策

## 状态
- **状态**: 已采纳 ✅
- **日期**: 2026-05-25
- **决策者**: 哲哥（需求方）+ 鼠小弟（技术执行）

---

## 背景

当前师资管理系统使用 **内存数据模型**（`teachers_full.py` 硬编码 + 运行时变量 `classrooms`/`teaching_sites`/`classes`），存在以下问题：

1. **数据不持久**：服务重启后，新增/修改的教室、现场教学点、班级全部丢失
2. **多人部署困难**：每个实例独立运行，数据无法共享
3. **数据导入依赖启动脚本**：每次启动都需要执行数据加载逻辑
4. **无法支持并发写入**：多用户同时修改时数据一致性无法保证

---

## 决策

### 方案选择：SQLite 文件持久化 + 启动时自动迁移

| 方案 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| **保持内存** | 启动极快，无需配置 | 重启丢失，无法扩展 | ❌ 放弃 |
| **SQLite 文件** | 零配置，单文件，Python内置支持 | 并发写入有限制 | ✅ 选择 |
| **MySQL/PostgreSQL** | 生产级，并发好 | 需要安装/配置/维护 | ❌ 哲哥目前不需要 |
| **Redis** | 键值快，适合做缓存 | 数据结构受限，需要额外部署 | ❌ 当前场景不适用 |

**决策理由**：
- 哲哥是软件开发小白，SQLite 零配置最友好
- 当前系统为单实例部署，SQLite 的并发限制不构成瓶颈
- 1848位教师数据 + 5291门课程已存在于 `teachers_full.py`，迁移成本可控
- 后续如需扩展，SQLite 到 PostgreSQL 的迁移路径清晰（SQLAlchemy 兼容）

### 实施策略

1. **数据库文件位置**：`instance/app.db`（Flask 惯例，.gitignore 排除）
2. **启动时自动检测**：
   - 如果数据库文件不存在 → 创建表结构 + 导入 `teachers_full.py` 数据
   - 如果数据库文件存在 → 正常连接，跳过导入
3. **数据导出备份**：提供 `/api/export` 接口，导出 JSON/CSV 备份
4. **迁移脚本**：`scripts/migrate_to_sqlite.py`，一次性将内存数据写入数据库

## 验证标准

1. 服务重启后，新增教室/现场教学点/班级不丢失 ✅

| 风险 | 应对 |

| 并发写入冲突 | 当前单实例部署无并发压力；后续如需扩展，迁移至 PostgreSQL |
| 哲哥不理解数据库文件 | 文档中明确说明 `.db` 文件的作用，不要手动删除 |
